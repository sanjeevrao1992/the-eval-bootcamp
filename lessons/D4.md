## Part 1: The Concepts

### Shape before depth

When you first open a production dataset, resist the urge to calculate metrics. Before computing anything, understand the shape of your data:

- How many rows do you have?
- What dimensions exist (categories, model versions, time periods)?
- What's the distribution across those dimensions?

This matters because shape determines what you can and can't say confidently. If 90% of your data is one category and 2% is another, you'll have strong signal on the first and almost none on the second. You haven't calculated anything yet, but you already know where your evaluation will be thin.

Shape also reveals design choices someone made before you arrived. If the dataset contains two model versions, someone is probably testing a new model. If timestamps cluster in a narrow window, this might be a snapshot rather than a representative sample. These aren't analysis results — they're context that shapes how you interpret everything else.

### Aggregates hide problems

Your team will hand you a single accuracy number: "The system is correct 78% of the time." That number is real. It's also nearly useless by itself.

Consider: an overall accuracy of 78% could mean any of these:

- Every category performs around 78% — uniform, no hidden problems
- One category is at 50% and everything else is at 90% — one bad segment dragging down the average
- The old model is at 66% and the new model is at 90% — the average masks a model quality gap

All three scenarios produce the same aggregate. They require completely different actions. In the first, you might ship. In the second, you'd hold one category and ship the rest. In the third, you'd deprecate the old model immediately.

The PM's instinct should be: **every time you see an aggregate number, slice it.** Ask: "78% for whom? 78% on what? 78% compared to when?" The story is always in the segments.

### pass@k: can the system get it right?

AI systems are non-deterministic. Run the same input through the system 5 times and you might get 5 different answers. This means a single test — "did it work?" — tells you almost nothing. You need to measure how the system behaves across multiple runs.

**pass@k** answers: "If I run the system k times on this input, did it get the right answer at least once?"

pass@5 = 1 means the system is *capable* of producing the right answer for this input. It got there at least once in 5 tries. pass@5 = 0 means it never got it right — this is a systematic failure, not bad luck.

To calculate pass@k across a dataset: count how many test cases had at least one correct run, divided by total test cases.

### reliable@k: does the system get it right consistently?

**reliable@k** answers: "If I run the system k times on this input, did it get the right answer every time?"

reliable@5 = 1 means the system is *dependable* for this input. Every run produced the correct answer. reliable@5 = 0 means at least one run was wrong — the system can fail on this input, even if it usually gets it right.

To calculate reliable@k across a dataset: count how many test cases had all runs correct, divided by total test cases.

### The gap is where production risk lives

The most important number in AI evaluation isn't pass@k or reliable@k. It's the gap between them.

If pass@5 = 95% and reliable@5 = 45%, that 50-point gap tells you: **the system can get almost everything right, but it only consistently gets about half right.** Spot checks will look great. Production will not.

Test cases in the gap — high pass@k, low reliable@k — are the system's "sometimes right" cases. They're the most dangerous because:
- They pass demo reviews ("look, it works!")
- They pass small-scale tests (often getting the right answer)
- They fail at scale (the 20–30% failure rate compounds across thousands of decisions)

Finding and characterizing the gap cases tells you exactly where to focus improvement efforts.

## Part 2: Exercise — From Accuracy to Reliability

Your ML team ran an A/B test comparing model v1 (current) against model v2 (candidate). They ran 20 representative menu verification test cases — each executed 5 times per model version — to measure consistency.

### Setup

Open `exercises/multirun-test.csv`.

Key columns: `test_id`, `restaurant_name`, `item_name`, `change_type`, `llm_decision`, `llm_confidence`, `llm_reasoning`, `ground_truth`, `run_number`, `is_correct`

Each test case appears in 5 rows (5 independent runs of the same input). 20 test cases total.

### Step 1: Understand the shape

Count: how many rows total? How many test cases per `change_type`? How many per `model_version`? Are any slices thin enough that you'd be cautious drawing conclusions from them?

### Step 2: The aggregate and its segments

Calculate overall accuracy (% of all runs where `is_correct` = TRUE). Then slice by `change_type`. Where does the aggregate mask a problem?

### Step 3: pass@5 and reliable@5

For each test case, calculate:
- pass@5: did the system get at least 1 run correct? (1 or 0)
- reliable@5: did the system get all 5 runs correct? (1 or 0)

Then aggregate across all 20 test cases: what % of test cases pass@5 = 1? What % reliable@5 = 1? What's the gap?

### Step 4: Characterize the gap

Which test cases are in the gap (pass@5 = 1 but reliable@5 = 0)? Look at their `change_type`, `llm_reasoning`, and the old/new values. What do the "sometimes right" cases have in common?
