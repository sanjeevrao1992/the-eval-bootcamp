## Part 1: The Concepts

### You can't review everything by hand

In D4, you sampled 10 failure cases and worked through each one. That was possible because it was a controlled exercise. In production, your system will make thousands of decisions per day. You need automated ways to check whether those decisions are correct — and different types of checks catch different types of problems.

The industry has converged on three types of automated quality checks, called **graders**. Each has a different cost, speed, and coverage profile. Understanding when to use which is one of the most practical skills in AI product management.

### Code-based graders: fast, cheap, and brittle

A code-based grader is a deterministic check written in code. It runs a rule and returns PASS or FAIL. No ambiguity, no judgment — just logic.

Examples for a menu verification system:
- **Threshold check:** if markup is above 25% and the system approved it → FAIL
- **Format check:** if the decision is not one of approve/reject/flag_for_review → FAIL
- **Consistency check:** if confidence is below 0.40 and the system didn't flag for review → FAIL
- **Contradiction check:** if the reasoning says "exceeds policy" but the decision is approve → FAIL

**Strengths:** fast (milliseconds), cheap (no API calls), 100% reproducible, easy to debug. You run the same check a million times and get the same answer.

**Weakness:** they can only catch what you can express as a rule. A code-based grader can check "is the markup above 25%?" but it can't check "is this reasoning fabricated?" or "does this description contain an unverifiable claim?" If the quality problem requires judgment, a code-based grader can't see it.

### Model-based graders: flexible but expensive

A model-based grader uses a second LLM to evaluate the first LLM's output. You give the judge LLM a rubric — specific criteria for what "good" looks like — and it scores the output.

Examples:
- "Is the reasoning grounded in the data provided, or does it cite information not in the input?"
- "Does the decision correctly apply the pricing policy stated in the context?"
- "Does the description change contain unverifiable sourcing or origin claims?"

**Strengths:** handles nuance and subjective criteria. Can evaluate open-ended text. Can assess qualities like "grounded," "consistent," and "complete" that are impossible to express as simple rules.

**Weaknesses:** non-deterministic (run it twice, you might get different answers), expensive (each evaluation is an API call), and it needs calibration. An LLM judge that hasn't been calibrated against human judgment will confidently produce wrong grades. The key question is always: does the judge agree with what a domain expert would say?

### Human graders: the gold standard you can't afford to scale

A human reviewer — ideally a domain expert — reads the output and judges its quality. This is the ground truth your other graders are trying to approximate.

**Strengths:** catches everything. A human can notice that "Free-range New Zealand lamb" is an unverifiable claim, that the reasoning contradicts itself, and that the confidence score doesn't match the decision quality — all at once.

**Weaknesses:** slow (minutes per case, not milliseconds), expensive (reviewer salaries), and — surprisingly — inconsistent. Studies consistently find human inter-rater agreement (measured by Cohen's Kappa) ranges from 0.2 to 0.5. Two reviewers looking at the same output will disagree 20–40% of the time. Humans get fatigued, apply different standards, and drift over time.

### Choosing the right grader: the decision framework

The three grader types form a pyramid:

**Code-based (bottom layer):** Catches clear rule violations. Use for everything you can express as a deterministic check. Runs on every single decision. Cost: near zero.

**Model-based (middle layer):** Catches nuanced quality issues. Use for criteria that require judgment but are well-defined enough to write a rubric. Runs on a sample, or on cases that pass code checks. Cost: moderate (API calls).

**Human (top layer):** Catches everything else. Use for calibrating your other graders, reviewing edge cases, and establishing ground truth. Runs on a small sample. Cost: high.

The practical instinct: **push as much as possible down the pyramid.** Every quality check you can express as code is one you never need to pay an LLM or a human to perform. But don't force it — a code check for something that requires judgment will produce false confidence.

## Part 2: Exercise — Evaluating the Evaluators

Your team has built a prototype quality-checking system for the menu verifier. They ran 20 representative decisions through all three grader types: a code-based checker, an LLM judge, and a human reviewer. Your job: figure out where each grader succeeds and fails.

### Setup

Open `exercises/grader-comparison-dataset.csv`.

Key columns: `eval_id`, `change_type`, `change_summary`, `llm_decision`, `llm_reasoning`, `ground_truth`, `code_grader` (PASS/FAIL), `code_rule` (which rule triggered), `model_grader` (PASS/FAIL), `model_critique` (the judge's reasoning), `human_grader` (PASS/FAIL), `human_note` (the reviewer's reasoning)

Each row is one menu verification decision, evaluated by all three graders.

### Step 1: Overall agreement

Calculate the agreement rate between each pair of graders: code vs human, model vs human, code vs model. Which automated grader aligns better with human judgment?

### Step 2: Where code-based graders fail

Find the cases where the code grader said PASS but the human said FAIL. What did the code grader miss? Look at the `llm_reasoning` and `human_note` columns.

### Step 3: Where code-based graders are too rigid

Find the cases where the code grader said FAIL but the human said PASS. Why did the code grader get it wrong? What would a more nuanced rule look like?

### Step 4: Design the grading strategy

Based on what you've seen: which quality dimensions should be checked by code, which by an LLM judge, and which reserved for human review?
